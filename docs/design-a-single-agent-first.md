# 方案 A:单 agent 优先(Claude Code 形态)——skill 默认在主循环展开

> 状态:提案,与 design-b-deliverable-channel.md 二选一(或分期)。
> 背景:loop 对 skill 结果的终答转写存在信息损耗(实测:sales-report
> 完整报表被压缩改写;详见两文档共享的问题定义一节)。

## 0. 问题定义(两方案共享)

skill 产出的**交付物**(报表/文档/盘点)经主循环终答转写后缺失、失真。
损耗链条:① skill 终答 → 主循环转写(两跳,各有压缩);② 超过
digest.over 的结果先被消化成摘要;③ L1 简洁性偏置。根源:**证据与
交付物两类结果共用一条"经模型嘴"的通路**。

## 1. 形态定义

把 Claude Code 的 skill 语义移植为默认形态:**skill 不再是子循环,而是
"指令注入 + 工具面解锁"**——主循环的模型亲自执行 skill 的步骤,做工作
的模型就是写终答的模型,转述边界消失。

```
现状:  用户 → 主循环 → [skill 子循环: 模型+工具 → 终答] → 主循环转写 → 用户
方案A:  用户 → 主循环(注入 skill 指令 + 解锁其工具)→ 主循环亲自执行 → 终答 → 用户
```

subagent(子循环)降级为**显式例外**,仅四种情况使用:
1. **并行扇出**(regions 多分支、批量盘点)——主循环无法并行推进多条思路;
2. **上下文隔离**(中间数据量大到会顶爆主上下文,如深度研究);
3. **专属模型**(component 需要与主循环不同的模型);
4. **治理边界**(某域需要独立预算/轮数/审批面)。

## 2. 机制设计

### 2.1 skill 的新执行形态:`use_skill` 展开

- 目录里的 skill 不再包装成"调用即子循环"的能力,而是注册为**过程卡**
  (procedure):`description` 仍进工具面供模型选择,但调用的效果是
  **返回 skill 的完整指令文本**(brief 渲染后)作为工具结果,同时把该
  skill 声明的工具子集解锁进主循环工具面(本轮生效)。
- 这正是 Claude Code Skill 工具的机制:skill 内容作为 tool result 进
  上下文,主模型照着做。skillpack(SKILL.md)天然适配——它本来就是
  给模型读的指令文档。
- 落点:`skill.Build`(skill/skill.go:112)增加 `mode: inline`(方案 A
  下为默认),产物 capability 的 handler 从"跑子循环"换成"渲染 brief +
  返回指令 + 标记工具解锁";react 引擎读取解锁标记扩展本轮工具绑定。

### 2.2 工具面管理(防爆炸)

主循环工具面 = 常驻工具 + 当前展开 skill 的工具子集。两级防线:
- **解锁制**:skill 的工具默认不进工具面,`use_skill` 展开后才可见
  (借鉴 CC 的 allowed-tools);轮次结束回收。避免 20 个 skill ×
  平均 3 工具 = 60 工具常驻的选择性灾难。
- **命名冲突**:跨 ns 同名工具经 `Rename`(capability.go:204)前缀化,
  现有撞名升级机制复用。

### 2.3 治理映射

| 现状(per-skill) | 方案 A 落点 |
|---|---|
| skill 级 max_rounds | 取消——主循环 max_rounds 统一管;耗尽收口引导已有(P2-5) |
| skill 级独立审批面 | 不变——审批本来就在工具粒度(GateApproval) |
| skill 级 Risk 聚合 | 改为工具级直接暴露(风险更透明) |
| 子循环 digest/truncate | 主循环同套 Ring 0,压力变大(见 §4) |

### 2.4 交付物保真在方案 A 下的形态

单跳消除了转述边界,但**终答仍是模型生成**——超长交付物在简洁性压力
下仍会被截。CC 的完整解法是文件通道;IM/HTTP 场景等价物仍然是
**attach 机制**(见方案 B §2)。诚实结论:方案 A 把损耗从"严重"降到
"轻度",要到"零"仍需 attach——两方案在这一点上不互斥。

## 3. 改动清单(硬切,pre-1.0)

1. `skill.Declaration` 增 `mode: inline|subloop`,方案 A 下缺省 inline;
   `components`(循环族)保留 subloop 语义不变(它们就是显式例外)。
2. react 引擎支持"轮内工具面扩展"(解锁标记经 TurnState 传递,
   WithTools 重绑)——这是最大的一块引擎改动。
3. namespace 装配:`skills:` 产物从子循环能力改为过程卡;`tools` 不再
   隐藏在 ns 内,随过程卡解锁(可见性规则重写)。
4. L1 增一节 skill 执行纪律(展开后按指令执行、完成前不得转做他事)。
5. 示例迁移:interactive 的 9 个 skill 改 inline,regions/deep-research
   保留 subloop(并行/隔离例外)。
6. 提示词行为变更 → MiniMax 全场景真机对照(必须,变更面大)。

## 4. 代价与风险(如实)

- **上下文膨胀**:所有中间结果进主上下文。interactive 旗舰场景(8 能力
  并行)在方案 A 下是 30+ 工具调用串行进主上下文,compaction 触发频率
  和摘要损耗都会显著上升——**损耗没消失,搬进了压缩层**。
- **并行能力退化**:主循环只有同轮 batch 并行;graph/rewoo 的结构化
  并行必须保留子循环形态,于是**两种心智模型并存**,配置面复杂度不降反升。
- **注入面扩大**:skill 指令直接进主上下文,外部 skillpack 的内容对主
  循环有了完整指令权(现状里它只能影响自己的子循环)。需要信任分级。
- **引擎改动大**:轮内工具面扩展在 eino react 上没有现成钩子,要绕
  MessageModifier + 重建 agent 或换自研循环,工程量是方案 B 的 3-5 倍。
- **多 agent 复用受损**:cap://skill 跨 agent 复用的"黑盒契约"消失,
  skill 的行为质量依赖宿主 agent 的模型与上下文状态。

## 5. 适用判断

方案 A 的甜区:**单 agent、任务中小、交互密集、模型较强**(CC 的场景
画像)。我们的目标画像(IM 接入、多域运营、批量并行分析、中档模型)
恰好多处踩在它的弱区。若采用,建议范围收窄为:给"轻 skill"(≤3 工具、
无并行、无隔离需求)提供 inline 模式作为**优化选项**,而非全局默认切换。

## 6. 与方案 B 对比速览

| 维度 | A(单 agent 优先) | B(子循环 + 交付物通道) |
|---|---|---|
| 交付物保真 | 单跳,轻度损耗仍在(长文仍需 attach) | 零损耗(原文直达) |
| 上下文成本 | 高(全过程进主上下文) | 低(隔离) |
| 并行 | 退化,需保留子循环例外 | 原生 |
| 工程量 | 大(引擎级) | 小(Ring 0 中间件 + 出站字段) |
| 心智模型 | 两种形态并存 | 现有模型不变,+2 个配置词 |
| 与业界对齐 | Devin/CC 路线 | LangChain artifact / A2A Artifact / Anthropic 研究系统路线 |
