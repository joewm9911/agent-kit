# 主循环"大脑质量"升级方案

> 状态:方案待 review。范围:主循环每轮的四件事——(1)提示词组织与记忆
> 唤醒、(2)任务拆解为 todo、(3)todo 随执行动态维护、(4)反思与目标达成
> 判断。从**效果**(任务成功率)出发,盘点现状、定位真差距、给可落地升级。
> 不含 eval(另见 eval-suite-design.md);但这几项的收益最终要靠 eval 量化。

## 0. 现状速览(四件事分别落在哪)

| 事 | 现有实现 | 位置 |
|---|---|---|
| 1 提示词组织 | 头部稳定(L1 规约+L2 persona+L3 环境,前缀缓存)+ 尾部变动(L4 记忆/计划/本轮问题重述) | `runtime/loop/prompt.go` PromptLayers.Modifier |
| 1 记忆唤醒 | autoRecall:长期记忆 kv 检索 + 窗外会话 bigram 词法召回,query=本轮原始输入 | `config/agent.go` autoRecall |
| 2 任务拆解 | todo_write(含正反例描述)+ 提示词推动"3 步以上先拆" | `todo/todo.go`、prompt.go loopPromptTodo |
| 3 动态维护 | 整表替换、每轮至多一次;计划每轮注入尾部(harness 强制可见);收口时 finish-check 拦未收口计划 | todo + review.go CheckedReviewer |
| 4 反思 | FinishReviewer(拦伪调用/空头承诺/叙述式执行)+ CheckedReviewer(计划未收口);reflection 是**子引擎**非主循环 | `runtime/loop/review.go` |

**关键架构发现**:review.go 的 `Reviewer` 框架(有界评审循环 + 有序评审器)
**正是第 3、4 件事的天然接缝**——finish-check 已经住在这里。加"目标达成
评审器"是顺纹理的做法,不是另起炉灶。

## 1. 提示词组织与记忆唤醒(高频效果,当前噪声偏多)

### 现状短板(从效果看)
- **召回不设相关性闸**:autoRecall 取固定 topK,**不管命中分高低都注**。低分
  命中稀释注意力——记忆注入越多,模型对当前问题的聚焦越差。对"效果"而言
  **精确率 > 召回率**:宁可少注一条,不可用噪声占注意力。
- **召回 query 太弱**:query 只用**本轮原始输入**。输入一简短("处理一下"
  "继续"),bigram 拿不到信号,召回全是噪声或空。
- **bigram 是词法召回**:只匹配字面,漏语义近邻。长期记忆的召回质量卡在这里。
- **无窗内去重**:注入的记忆可能和最近几轮消息重复,重复占上下文。

> 范围收敛:本轮**先只看 session 召回**(窗外会话历史),长期记忆(kv/向量)
> 后面单独看。bigram 打分实测是**字符 bigram 重叠原始计数**(impl/session/
> bigram,未归一),纯词法——**不足以做可靠的精细相关性闸**(会丢"相关但换
> 说法"、留"字面撞但无关")。所以 U1.1 从"给 bigram 拧阈值"改为下面三项
> **不需要 embedding** 的启发式,它们对 session 场景性价比远高于换算法。

- **U1.1 短/指代查询近因兜底(补最致命的死角)**:"再查那个订单""上面说的"
  这类回指查询和目标轮次几乎零词法重叠,bigram 结构上救不了。当 query 内容
  bigram 很少(如 <3)或含指代词(那个/它/刚才/上面)→ 不靠词法,**回退到
  最近滚出窗口的 N 轮**。对话指代八成指近处,近因兜底覆盖大部分。
- **U1.2 近因加权**:召回分 = 词法分 × 距离衰减。对话相关性随距离衰减是强
  信号,当前 bigram 纯按重叠计数排序,一个远古高重叠会压过近期中等相关——
  纯算术改动,排序立刻更对。
- **U1.3 重叠归一化**:overlap 改 Jaccard 式(/并集 或 /min 长度),消除长
  消息虚高(原始计数偏袒长 assistant 回答)。一行改动。
- **U1.4 截断与执行记录两个硬编码限制**:召回每条**截 120 字**、且**只扫
  user/assistant 跳过执行记录**——都写死在 bigram 里、无配置旋钮。就算匹配
  100% 准,能送进 prompt 的内容也被这俩卡住。两个限制做法不同,详见下节。

#### U1.4 落地细化

**限制 1:120 字截断——真问题是"截错地方",不是"太短"**
现在 `clip(content, 120)` 取**前 120 字**;但 bigram 匹配点常在消息**中间**,
于是展示的前缀里**根本不含匹配到的内容**——定位对了、送出的片段却答非所问。
做法(按价值序):
- **U1.4a snippet-around-match(核心,纯改进、默认启用)**:改成返回**命中
  bigram 密度最高区段为中心的窗口**(搜索结果摘要式 `…命中上下文…`),而非
  前缀。这一步修的是"片段不含匹配"的隐性 bug,比调长度重要。
- **U1.4b 长度可配**:120 提为 `retriever_config.snippet_len`,缺省 ~160。
- **U1.4c 总量预算(可选)**:top_k 条共享字符预算,防"3 条各 300 字"撑爆
  尾部;宁少而准。

**限制 2:排除执行记录——选择性纳入,三重护栏,opt-in**
数据已在手边:autoRecall 传给 bigram 的 `older` 本就含执行记录(RawHistory
保全部原文),bigram 只是按 role 主动跳过——所以"纳入"是 bigram 内**放宽
role 过滤**,零上游管道改动。护栏:
- **U1.4d `retriever_config.include_tools: false`(默认关)**:等同现状,
  需要事实回指的场景(ops)才开;
- **只纳入紧凑形态**:`record_tools: summary` 存的执行记录本就是摘要版;
  绝不纳入原始大输出。命中带 read_result 指针的**附指针**(召回给线索,
  全文模型自己翻);
- **单独限额**:工具命中最多 1~2 条,与对话命中分开计,防淹没对话召回。

**评估方法(eval 后置的替代)**:定向真机点检——构造 3~5 个回指场景(第 N 轮
答案是工具事实,第 N+M 轮追问),改前/改后 A/B 看①正确事实是否被召回、②注入
字符数是否失控。方向足以判定,不等完整 eval。

**落地边界**:全在 `impl/session/bigram`,schema 只加 `retriever_config`
透传;默认行为不变(around-match 纯改进、include_tools 默认关);实现前先确认
`TrajectoryMessage` 的 role,决定放宽过滤怎么写。**先做 U1.4a(无配置、无风险、
默认启用),U1.4d 的 include_tools 作 opt-in 增量跟后。**

### 登记(session 之外,后面单独看)
- **语义召回(vector)**:词法/近因都救不了的深层语义近邻才是向量的地盘,
  但 session 场景近因兜底已覆盖大半,性价比不高;留给长期记忆一并看。
- **任务类型自适应**:scan vs 审批注入不同领域线索——收益真实但依赖 eval
  校准,不宜盲调。

## 2. 任务拆解为 todo(当前尚可,真提升依赖度量)

### 现状
todo_write 描述质量已经不错(何时用 + 正反例),提示词也推动拆解。计划每轮
注入、harness 强制可见——这块地基是好的。

### 短板
- 拆不拆**全凭模型自觉**,没有"该拆没拆"的信号。终简多需求输入下,模型可能
  直接埋头做、跳过计划。
- 拆解**粒度**好坏没有反馈(拆太粗=没意义,太细=噪声)。

### 升级
- **U2.1 turn 起始的拆解决策强制**:Focus 指令已提到 todo_write,强化为
  "多部分/3 步以上输入,先显式做一次'要不要拆'的判断再动手"。低成本。
- 粒度质量的改进本质上要靠 eval 度量(plan-appropriateness 指标),**登记**。
  这块不宜盲调——现状可用,砸开 eval 后再针对性调。

## 3. todo 随执行动态维护(有接缝,当前只在收口把关)

### 现状
整表替换 + 每轮至多一次;计划每轮注入(可见);finish-check 在**收口时**拦
"还有 pending 却想结束"。

### 短板
- 唯一强制点在**收口**。执行**中途**的漂移不被主动纠:模型做完一项忘了标
  completed、或发现新工作没加进计划——要拖到收口才可能被 finish-check 逮到,
  且 finish-check 只看"有没有 pending",不看"completed 的项有没有真做"。
- "claimed done but no evidence":标了 completed 但本轮没有对应工具调用支撑
  ——当前不查。

### 升级
- **U3.1 计划漂移提醒**:检测"连续 N 轮有工具调用但计划没更新"→ 尾部注入
  一句"把计划和你已做的对齐一下"。轻量、不打断。
- **U3.2 finish-check 加强**:收口时除了"无 pending",再查 completed 项是否
  有工具证据(本轮/本任务有对应调用)。标了完成却零证据 → 弹回要求补做或
  据实改状态。这是把"诚实执行"纪律从"没伪造调用"延伸到"没伪造完成"。

## 4. 反思与目标达成(**最大差距**,接缝现成)

### 现状(诚实说)
主循环的两个评审器查的是**执行诚实**:FinishReviewer 拦伪调用/空头承诺,
CheckedReviewer 拦计划未收口。**没有一个查"目标达成"**。reflection 只是
子引擎,主循环不带。

后果:模型可以**收齐所有 todo、给一个看似合理、却答偏了原问题**的终稿,
当前**无人拦截**。多部分问题只答了一半、扫描任务漏了关键实体、退款处理没真
改状态却说"已处理"——这些都能溜过现有 finish 检查。这是四件事里对成功率
影响最大的空洞。

### 升级(核心)
- **U4.1 目标达成评审器(GoalReviewer)**:在 review.go 框架里新增一个评审器
  ——收口前拿终稿对照**原始目标**(runctx.Input + 活跃计划),判"这个答案是否
  真的覆盖了问题的每一部分"。不达成 → 以"缺了 X、Y"弹回,受同一全局重试预算
  约束。这就是你要的主循环反思,而且 review.go 就是为它准备的接缝。

  两种实现,按成本选:
  - **轻量(推荐先做)**:结构化自检——收口前给模型追加一条"对照原始目标逐条
    核对你的答案,列出未覆盖项;若全覆盖回 DONE"的 system,让**同一模型**自评
    一次(不额外起 judge)。命中未覆盖项就继续,否则放行。
  - **重(可选)**:独立(可不同 provider)judge 打分,更准但每次收口一次额外
    调用。
- **触发收敛成本**:只在"有计划的多步任务"或"多部分输入"上触发目标核对,
  单一简单问答跳过——避免给每个"查个库存"都加一轮自检。
- **与 grounding 联动**:目标核对里顺带查"答案里的关键结论有没有工具证据"
  (反幻觉),和 eval 的 grounding 指标同源。

## 5. 上下文压力:最近长消息的泄压阀(与 CC 对齐)

### 现状(按消息类型不同)
- 最近的长**工具结果** → digest(消化 + read_result 指针)在调用当时就压,
  原文可取回。**已处理**。
- 滚出窗口的**老消息** → compaction 滚动摘要。**已处理**。
- 最近的长 **user/assistant 文本** → **无策略**:窗口按条数不按大小算,
  compaction 明确保护 keep_recent 不碰最近,tool_clear 也只清工具、且保护最近。
  长用户粘贴/长 assistant 回答原样堆着,直到滚出 keep_recent 才被摘要。

### 与 CC 的差异(诚实对照)
CC 也没有"最近长消息压缩器"——它靠:①工具/文件输出**源头截断**(Read 2000
行上限、Bash 上限,引导用 grep/精确读),对应你已有的 digest;②**大上下文
窗口**吸收长近期消息;③**临近上限的整体 auto-compaction**——这一步**会压到
最近**。关键差异:agent-kit 的 `keep_recent` 是**硬保护**,压力下也不动最近,
**没有泄压阀**;CC 的临限压缩该压就压(含最近)。

### 升级
- **U5.1 上下文压力泄压阀**:当总量逼近上限(max_tokens 触发)、且膨胀源就是
  最近窗口时,允许 compaction **临时突破 keep_recent 保护去压最近**;或对
  **超长的最近单条** user/assistant 做 digest 式消化(要点入上下文 + 原文进
  result store 带指针)。只在真顶到窗口时才动最近,平时仍保护近期连贯性——
  比"无脑摘要最近几条"更对。低频触发,不影响常态。

| 序 | 升级 | 为什么这个序 |

| 2 | **U1.1 近因兜底 + U1.2 近因加权 + U1.3 归一化**(session 召回) | 高频、每轮生效、零延迟;补回指死角 + 排序更对;不需 embedding |

## 6. 与 eval 的关系(为什么顺序是"先做 U4.1/U1.1,再上 eval 量化")

U4.1(目标核对)、U1.1(相关性闸)、U3.2(完成证据)都是**第一性可辩护**的
机制改进——减噪声、拦漏答、防伪完成,不需要 eval 就知道方向对。但它们**改多少
才最优**(阈值、触发条件、自检提示词措辞)必须靠 eval 数字调。所以:先落这三个
机制(结构对),再用 eval 把参数拧到最优。两者不互斥,eval 是这些升级的标尺。

## 8. 需裁决
1. U4.1 用**轻量自检**(同模型多一轮,省成本)还是**独立 judge**(更准更贵)?推荐先自检;
2. 目标核对的**触发面**:仅多步/多部分任务(推荐)还是所有非平凡任务;
3. U1.1 相关性阈值先给经验值(如 bigram 分 > 0.3)第一轮只观测不硬丢,还是直接启用丢弃;
4. vector 召回本批做还是登记(取决于长期记忆用量是否已成痛点)。
