## 0. 总体评估

**架构底子是好的**:分层守卫、注册表 init 只读、Ring 0 中间件次序与注释一致、
兄弟实现(session/store/memory 三后端)语义明显是对照着写的、错误消息普遍
可执行、Ref/Outbound/redisconn 等契约文档达到 OSS 水准。审计确认干净的清单
见 §6。

**三个系统性缺陷模式**(单点修复之外要修"模式"):

1. **静默吞配置**:YAML 无严格模式(typo 键消失)+ 裸 map 断言(docker
   timeout/memory、redis db、store dir)+ 静默枚举回退(record_tools/
   session_mapping)+ 静默丢弃(sampling params、memory.seed、retry)。
   这一类直接架空了框架自己最引以为豪的 fail-fast 哲学。
2. **双装配路径漂移**:`Build`(单文件)与 `BuildApp`(多文件)四处不同步
   (texts / HTTP suspend / legacy 校验 / logger),证明"同一逻辑两份实现"
   必然漂移,需要共享内核 + 漂移测试。
3. **TurnTerminal 信号被工具错误兜底吞掉**:审批门、direct、rewoo 三处把
   挂起/中断/预算硬停错误转成结果字符串——挂起模式下审批功能性坏死。
   react 有正确的 `turnTerminal` 判定,其余路径没抄。

## 2. P1 —— 高价值批次(建议按批推进)

### 批 A:文档大修(全部 [doc],成本低、当前对新用户是硬伤)

- README 旗舰示例**装配不过**:`max_steps`(2 处)、`memory.tools` 均是会
  fail-fast 的死键(README:224/232/421);
- **输入模型未文档化**:`{$user_input}` 全 README 缺席、`{$input}` 还写着
  "用户本轮原话"(已是作用域输入)、`input:` 步骤字段无任何用户面文档——
  P1-P5 的核心成果没进 README;
- README 说 window<keep_recent"能跑不报错",代码**装配期硬报错**(agent.go:321);
- interactive/main.go 头注释与代码**反着**(默认 MiniMax 写成默认智谱);
- ~20 条 README 链接指向重构前路径(loop/、capability/ 等);
- CHANGELOG **无 Breaking 段**(P3/P4/几个改名硬切全部未记——自家政策要求有);
- README"跨 ns 只能引 skill"过时(component export/imports 已存在);
- `workflows:` 平铺段不存在;prompt ref 语法三处文档两处错(含教已移除语法);
- docs/model-input-surface.md 已被实现超越但无横幅,仍被引用为"现状";
- 各处残留 `work_dir` 措辞;smoke README 缺 regions;CONTRIBUTING 指向不存在的
  extending.md。

**方案**:一个文档批一次清完,并加 CI 链接检查(防再烂)。**成本:低-中。**

### 批 B:配置解码与校验统一(修"静默吞配置"模式)

- 裸 map 断言全部改 `decode.Config`(docker timeout/memory、store/file dir、
  session/file、redisconn db 字符串静默进 DB 0);
- 静默枚举转 fail-fast:`record_tools`、`session_mapping`(typo 把群成员并进
  共享会话,隐私相关)、`approval.mode`、`reply_mode`;
- `memory.write` scope 非法值静默按 user → 报错;
- `ParamsSchema` 未知 type(`int`/`str`)静默转 string → 装配期报错;
- kind-`*` 通配 ParseRef 收下但永远匹配不到 → 拒绝;
- 模型 sampling params(temperature 等)静默丢弃 → 透传或报未知键;
- `memory.seed` 在无 expose_tools/recall 时静默丢弃 → 修条件或报错;
- `reliability.retry` 在不建专属模型的层级静默无效 → 接线或文档明示;
- `default_model` 无 legacy 重定向(其余改名都有)。

**方案**:一个"配置严格化"批;与 P0-5 严格模式同批做掉最经济。**成本:中。**

### 批 C:子循环一致性(agent 与 skill 中间件栈对齐)

- 子循环缺 `ControlTools`(中断/插话只能靠 ctx-cancel,steer 排队到 skill 返回);
- `TruncateResults(caps, 0)` 硬编码默认,agent 配置的 truncate 不透传;
- `todo: true` 的 skill 缺 `FinishCheck`(开了计划却没有收口纪律——恰是最需要的地方);
- **"只有最终消息返回给调用方"契约行只在 todo 变体里**,NoTodo 变体(组件默认)
  没有——而空壳病灶恰在组件;把这行移到 loopPromptTail(它是循环力学事实,
  不是 todo 特性);
- 子 agent scope 无调用序号(`sub:name` vs `comp:name#N`),同轮两次调用共享
  dedup/denied 计数;
- 混合语言的 harness 消息(timeout 中文 vs react 英文)统一。

**成本:中。**其中契约行迁移按家规做一次 MiniMax 快速对照。

### 批 D:httptool/feishu/dedup/compaction 小修集

- httptool:未声明参数被透传后端(注入面,声明即白名单)+ required 不校验 +
  路径占位符未消费残留 + `io.LimitReader` + rune 安全截断;
- feishu:`call()` 不查 HTTP status(5xx 空 JSON 当成功)+ 卡片大小护栏
  (超 30KB 流式更新永久失败)+ token 刷新 single-flight;
- dedup:对 `TagInteractive` 豁免(第三次相同 ask_user 被拦,故意再确认不可能);
- compaction tool-clear 误清 digest-failure 指针消息(`[结果过长且消化失败`
  前缀未豁免,可取回指针被换成"重新调用");
- 结构化输出失败丢整轮会话(agent.go:196,与失败落痕政策相悖)+ structured
  自带弱 JSON 提取器(该复用 ExtractJSON);
- thinkstrip flush 重复发 ToolCalls(窄触发);registry 错误路径锁外遍历 map
  (session/recall/memory/model/engine 五处,竞态检测器地雷);
- 审批 recall 空值 raw[0](已修)之外:CLI interactor 忽略 ctx(取消不了 stdin
  读,泄漏持锁 goroutine)。

**成本:每项低,打包一批。**

### 批 E:记忆质量(effectiveness 主战场)

- Search 返回 `map[string]string`:无排序、跨 scope 同键碰撞(**shared 覆盖
  user,方向反了**)、超 limit 时**每次随机子集**(prompt-cache 敌对、eval 敌对);
- 整句子串匹配打不中多关键词查询(工具描述却教模型"by keyword");repo 自有
  bigram Jaccard 没用在这里;
- redis 每次 Search 全量 HGetAll;
- 召回 memo 跨轮不过期(同字面输入复用旧召回)。

**方案**:Search 契约改 `[]Hit{Key,Value,Score}`(破坏性,pre-1.0 硬切)+
user-first 合并 + bigram 打分;**成本:中,收益直接作用于效果。**
